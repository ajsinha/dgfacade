Glossary of Terms

Comprehensive reference of all terms and concepts used in DGFacade

Quick Navigation
A B C D E F H I J L Q R S T

A

ADMIN Role

The highest privilege level in DGFacade. ADMIN users have full access to all system features including user management, broker configuration, channel monitoring, and handler execution. Configured per user in config/users.json.

Security Roles
API Key

A token-based authentication credential for programmatic REST API access. API keys allow applications to invoke handlers and query status without username/password authentication. Managed per user in config/users.json. Passed via the X-API-Key HTTP header.

Security API
ActiveMQ

An open-source message broker that supports JMS (Java Message Service). DGFacade can connect to ActiveMQ as a broker type to consume messages from JMS queues and topics. Configured via JSON files in config/brokers/.

Broker JMS

B

Backpressure

A flow-control mechanism that prevents memory exhaustion when consumers are slower than producers. In DGFacade, each channel maintains an internal bounded queue. When queue depth reaches the critical threshold (default 90%), the broker consumer is paused until the queue drains below the resume threshold (default 70%). This propagates slowdown signals back to the broker without losing messages.

Channel Queue Performance
Broker

An external message system that DGFacade connects to for data retrieval. Supported types include Kafka, ActiveMQ, RabbitMQ, and IBM MQ. Each broker is defined by a JSON file in config/brokers/, where the filename becomes the broker's logical name. Channels reference brokers by this name.

Infrastructure Configuration
Broker Config Directory

The filesystem directory scanned at startup for broker JSON files. Default: config/brokers/. Configurable via dgfacade.brokers.config-dir in application.properties.

Configuration

C

Channel (Input / Output)

A logical data pipe that connects a broker to DGFacade via destinations (topics, queues, paths, tables). Input Channels subscribe to destinations and consume messages into DGFacade (fan-out). Output Channels publish messages from handlers to destinations (fan-in). Defined in config/input-channels/ and config/output-channels/.

Core Concept Configuration
Channel Config Directories

Input channels: config/input-channels/ (property: dgfacade.input-channels.config-dir). Output channels: config/output-channels/ (property: dgfacade.output-channels.config-dir).

Configuration
Critical Threshold

The internal queue depth percentage (default 90%) at which the broker consumer is paused. When the queue fills to this level, backpressure is fully engaged to prevent memory exhaustion. The consumer resumes when depth drops below the drain resume threshold.

Backpressure

D

DGFacade

Data Gateway Facade — a high-performance data gateway that acts as a facade between external message brokers and internal application handlers. It provides JSON-configured broker connections, channel-based data routing, fan-out distribution, and backpressure management.

Product
Drain Resume Percentage

The internal queue depth percentage (default 70%) below which a paused broker consumer is resumed. This creates a hysteresis band between the critical threshold and resume threshold to prevent rapid pause/resume oscillation.

Backpressure

E

Execution Engine

The core processing component that dispatches incoming messages to registered handlers based on request type. It manages the handler lifecycle: registration, dispatch, execution, state tracking, and error handling. Handlers are discovered via Spring component scan at startup.

Core Component Handler

F

Fan-Out

The message distribution pattern used by DGFacade channels. When a message is dequeued from the internal queue, it is delivered to ALL registered listeners for that channel topic/queue. Each listener receives its own copy. If there are N listeners, each message results in N deliveries. Failures in one listener do not affect others.

Distribution Core Concept
FileSystem Channel

A channel type that watches a filesystem directory for incoming data files. When new files matching the configured pattern appear, they are read and their contents placed into the internal queue for fan-out distribution. Does not require a broker reference.

Channel Type

H

Handler

A pluggable processing component registered with the Execution Engine. Each handler declares the request types it supports. When a matching message arrives (via channel listener or REST API), the handler processes it and produces a response. Handler execution is tracked with timing, status, and error information.

Core Concept Processing
Handler State

A record of a handler's execution. Includes fields like handlerId, requestType, status (SUCCESS, FAILURE, RUNNING, TIMEOUT), startTime, durationMs, and errorMessage. Viewable in the monitoring dashboard.

Monitoring

I

Internal Queue

A bounded in-memory queue (backed by java.util.concurrent.LinkedBlockingQueue) that sits between a channel's broker subscription and the fan-out engine. Its configurable depth is the primary mechanism for backpressure. When the queue fills, the broker consumer is paused. When it drains, the consumer resumes.

Channel Backpressure
IBM MQ

IBM's enterprise messaging middleware. DGFacade supports IBM MQ as a broker type, connecting via queue manager, channel, and optional SSL. Commonly used for mainframe integration in financial services.

Broker

J

JSON Configuration

DGFacade uses JSON files for broker and channel configuration. Brokers are in config/brokers/*.json and channels in config/channels/*.json. The filename (minus extension) becomes the entity's logical name. This file-based approach allows configuration changes without code modifications.

Configuration
JMS (Java Message Service)

A Java API for message-oriented middleware. DGFacade uses JMS to connect to ActiveMQ and IBM MQ brokers. Channel type jms indicates a JMS-based channel that can subscribe to JMS queues and topics with features like durable subscriptions and message selectors.

Protocol Channel Type

L

Listener

An object registered to receive messages from a channel's fan-out engine. Listeners are typically handlers or logging/auditing components. Multiple listeners can be registered on the same channel topic/queue, and each receives every message. Listeners have independent error handling and retry policies.

Fan-Out Core Concept

Q

Queue Depth

The number of messages currently buffered in a channel's internal queue. Monitored as a percentage of the queue's maximum capacity. Queue depth is the primary signal for backpressure: NORMAL (0-70%), WARNING (70-90%), CRITICAL (90-100%). Viewable on the monitoring dashboard and via the REST API.

Monitoring Backpressure

R

RabbitMQ

An open-source message broker implementing the AMQP protocol. DGFacade supports RabbitMQ as a broker type with virtual host, exchange, and queue configuration. Configured via JSON files in config/brokers/.

Broker
REST Channel

A channel type that creates HTTP endpoints accepting POST requests. Messages submitted to these endpoints are placed into the internal queue for fan-out distribution. Does not require a broker reference. Useful for webhook ingestion.

Channel Type API
Request Type

A string identifier that links incoming messages to the appropriate handler. When a message arrives, the Execution Engine examines its request type and routes it to the registered handler. Each handler declares which request types it supports during registration.

Handler Routing

S

SSL/TLS

Encryption protocol used to secure broker connections. All broker types in DGFacade support SSL/TLS via an ssl block in the broker JSON file. Configuration includes keystore path/password (client certificate), truststore path/password (CA certificates), protocol version, and cipher suite.

Security Broker
Subscription

A channel's declaration of which topics or queues to listen to on its referenced broker. Each subscription specifies a name (topic/queue name), type (topic or queue), and optional filters like JMS selectors or Kafka pattern subscriptions. Messages from all subscriptions feed into the channel's internal queue.

Channel Configuration

T

Topic

A named publish-subscribe destination on a message broker. Multiple consumers can subscribe to the same topic and each receives a copy of published messages. In DGFacade, channels subscribe to topics via their subscriptions array. Kafka topics and JMS topics are both supported.
