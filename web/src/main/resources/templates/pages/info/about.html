DGFacade

A high-performance, configuration-driven data gateway facade built on Apache Pekko actors and Spring Boot. Multi-channel request ingestion, pluggable handler dispatch, dynamic proxy support, SSL/TLS with PEM certificates, and enterprise-grade messaging across 6 broker types.

Version 1.2.0 Production Ready Patent Pending

The name DGFacade combines DG (Data Gateway) with Facade — a unified interface that conceals the complexity of handler dispatch, actor orchestration, multi-broker messaging, and authentication behind a clean, consistent API. Inspired by the GoF Facade Pattern, DGFacade lets consumers interact with a single gateway while sophisticated routing, TTL enforcement, and result delivery happen transparently.

Multi-Channel Ingestion

Accept requests via REST API, WebSocket, Kafka, ActiveMQ, RabbitMQ, or IBM MQ. Every request is normalized into a canonical DGRequest for uniform processing.

Pluggable Handlers

Drop-in handler JARs for custom request types. Implement the DGHandler interface or use any POJO — the dynamic proxy wraps it automatically.

Pekko Actor Engine

Apache Pekko actors for concurrent, resilient handler execution. Millions of handlers can run simultaneously with per-request TTL enforcement.

Configuration-Driven

JSON-based handler, broker, channel, user, and API key configuration. Hot-reload support — no server restart required for config changes.

Dynamic Proxy

Register any POJO as a handler — DGHandlerProxy discovers init/process/stop/cleanup methods via reflection and wraps them into the DGHandler lifecycle.

SSL/TLS & PEM

Full SSL/TLS support with PEM certificates, JKS/PKCS12 keystores, and mTLS. Auto-detects format and handles PKCS#1, PKCS#8, and EC private keys.

6 Broker Types

Kafka, ActiveMQ, RabbitMQ, IBM MQ, FileSystem, and SQL — all with unified publish/subscribe abstraction and SSL support.

Input Fan-Out Channelsamp; Output Channels

Input Channels (fan-out, subscribe) and Output Channels (fan-in, publish) with destinations and backpressure. Decouple producers from consumers with bounded queues.

Streaming Handlers

Handlers can emit multiple streaming updates via WebSocket. Sequence-numbered partial results with final completion signals.

Phased Startup

Orchestrated startup and graceful shutdown with dependency ordering. Each subsystem initializes in sequence for guaranteed consistency.

API Key Authentication

Secure programmatic access with API keys. Per-key request type restrictions, rate limiting, and user resolution from the API key.

Web Dashboard

Real-time monitoring dashboard with handler states, execution metrics, user management, and comprehensive help documentation.

Prometheus & Grafana

Competitive Advantage

Why DGFacade Outperforms Alternatives

vs Spring Integration
  • Actor Concurrency — Pekko actors vs thread pools
  • TTL Enforcement — per-request timeout with auto-suspension
  • Dynamic Proxy — register any POJO as a handler
  • JSON Config — no XML, no annotations, pure config files
  • Built-in UI — monitoring dashboard included
vs Apache Camel
  • Simpler Model — request → handler → response, no route DSL
  • Lower Footprint — no 300+ component dependencies
  • User Isolation — per-user handler configs and API keys
  • PEM Support — native PEM certs without Java keytool
  • Handler Streaming — native streaming response support
vs MuleSoft / Kong
  • No Licensing — single JAR, no per-node fees
  • Embeddable — runs as Spring Boot app, not a platform
  • Custom Logic — full Java handlers vs flow designers
  • Multi-Broker — 6 broker types with unified API
  • Self-Hosted — full control, no vendor lock-in

Architecture

System Components
  • Execution Engine — Pekko actor-based handler dispatch & TTL management
  • Handler Supervisor — Actor lifecycle, crash recovery, backpressure
  • DGHandlerProxy — Reflection-based dynamic proxy for POJO handlers
  • REST API — Spring Boot REST & WebSocket endpoints
  • Messaging Layer — Kafka, ActiveMQ, RabbitMQ, IBM MQ, FileSystem, SQL
  • SSL Helper — PEM, JKS, PKCS12 context builder with mTLS
  • Config Registry — JSON-based handler and broker config with hot-reload
  • Channel Manager — Input (fan-out) and Output (fan-in) channels with destinations and backpressure
  • User & API Key Service — Authentication, role-based access, rate limiting
